# Phase 03 — Onboarding & calendar sources

> **Goal:** everything that gets a calendar onto the device. This establishes **identity** (= calendar subscription tokens) and the durable storage of it — the thing the whole app hangs on.
>
> **Depends on:** Phase 02 (server-read pattern, storage). **Modules:** `onboarding`, `add_school`, `school`, `school_group`, `qr_code`, `import_ical`.

## Rough steps

1. ✅ **Onboarding flow** — landed (`add-mobile-onboarding-flow`, ADR [015](../../../docs/mobile/architecture-book/decisions/015-onboarding-flow-shape.md)): a **native-default** brand/welcome surface as the onboarding entry over the existing school step (Expo Router stack, welcome-first). The **designer-polish artifacts** (real illustrations / final brand copy / optional motion / the white-on-brand-CTA `primaryStrong` decision) are inboxed (HUMAN: `inbox/2026-06-15-onboarding-design-polish.md`); the on-device DoD axes are inboxed (`inbox/2026-06-15-onboarding-flow-dod-manual.md`).
2. ✅ **School / school-group selection** — landed (`add-mobile-school-picker`, ADR [016](../../../docs/mobile/architecture-book/decisions/016-school-group-multi-select-commit.md)): grew Phase 02's read path into the **full picker** — the group step is now **multi-select with an explicit confirm-commit** (toggle leaves into a pending set, confirm persists the whole set through the unchanged identity-only store, empty confirm guarded), school search is **accent-insensitive name-or-code** via a pure `data/search.ts` helper, and completion **dismisses the whole onboarding Stack** (`router.dismissTo`). No new dependency / native change. On-device DoD axes inboxed (extended the existing `inbox/2026-06-14-school-selection-dod-manual.md`).
3. ✅ **QR scan** — landed (`add-mobile-qr-scan`, ADR [017](../../../docs/mobile/architecture-book/decisions/017-qr-scan-camera.md)): `expo-camera` barcode scanning (replaces `mobile_scanner`), behind the repo's first `src/features/calendar-sources/` folder (the shared QR · iCal · persistence cluster) — a pure 90%-gated scanned-value parser (`data/`, mirrors the Flutter `QrCodeResult.url` wire format) feeding a `CameraView` scanner screen (`ui/`) with permission request/denied states; `recordAudioAndroid: false` so no `RECORD_AUDIO` leaks. The camera can't be CI/Maestro-driven — the token→state wiring is Jest-proven, the on-device camera + `expo prebuild --clean` permission proof inboxed (`inbox/2026-06-15-qr-scan-dod-manual.md`).
4. ✅ **iCal import** — landed (`add-mobile-ical-import`): grew `calendar-sources/` with a URL-entry import (Flutter `import_ical` parity — **URL-only, server-POST** via the committed generated `POST /calendars` over `customFetch`, the server owns `.ics` parsing). A pure lenient URL validator (`data/validate-url.ts`, localizable keys), accessible loading/error/retry, a welcome CTA; QR + iCal converge on one success handoff. Growth within ADR 017 (no new ADR/dep/native change). On-device round-trip + Crashlytics arrival inboxed (`inbox/2026-06-15-ical-import-dod-manual.md`).
5. ✅ **Identity persistence** — landed (`add-mobile-calendar-identity-persistence`, ADR [018](../../../docs/mobile/architecture-book/decisions/018-user-calendar-storage.md)): a durable **Drizzle** `user_calendars` token store (MMKV genuinely weighed and rejected — relational identity + Phase-09 importer target + free migration-runner durability), the schema mirroring the Flutter `toDbMap()` **verbatim** (importer fidelity — `id`-is-record-key, `token` the irreplaceable identity, dates TEXT ISO-8601 UTC, `visible` boolean), the second real migration committed, the full `data/user-calendars/` layer (mappers + repository + reactive hook + the shared `addCalendarFromUrl` persist seam), and ships 3/4's success paths rewired to persist durably (the ephemeral holder removed). On-disk restart/kill/cache-clear durability is the on-device manual pass (HUMAN: `inbox/2026-06-16-calendar-restart-durability.md`). **The Phase-09 importer-target schema is in place to receive the recovered `user_calendars.token`.**
6. 📥 **Confirm Android storage** — needs physical Android hardware, so it's a HUMAN inbox item, not a ship (`inbox/2026-06-15-android-storage-verification.md`): verify the open Android items from the [migration research §6](../00-exploration/data-persistence-migration.md#6-device-verification-done) (prefs backend + sembast path) on a real Android device. It gates the Phase-09 importer, not any Phase-03 feature.

## Exit criteria

- ✅ A fresh user can add a calendar via **school pick** (steps 1–2), **QR** (step 3), and **iCal** (step 4) — built, typed, lint-clean, unit/component-tested, i18n FR/EN, a11y. **Maestro E2E green on BOTH iOS and Android** (commit `26ba25d`, run 27582517604: onboarding + ical-import + personal-events + settings flows all pass; the launch runs `runMigrations()`, so the migration materializes on-device without crash). The per-feature on-device DoD axes (VoiceOver/TalkBack, contrast, touch-target, native-feel, real camera scan / real `.ics` round-trip) are inboxed for a human device pass (each feature's `*-dod-manual.md`). _(The Android E2E `:app:packageRelease` build failure first seen was a transient CI-runner packaging flake — a no-code-change re-run went green.)_
- ✅ Calendar **tokens persist across restarts** — durable Drizzle `user_calendars` store (step 5), restart-simulation + repository + mapper-fidelity Jest tests green; **schema is migration-ready** (mirrors the Flutter wire format verbatim, the Phase-09 importer target). On-disk restart/kill/cache-clear durability is the inboxed on-device pass (`inbox/2026-06-16-calendar-restart-durability.md`) — no list UI ships this phase, so there is no observable surface for a Maestro restart assertion; the Jest restart-simulation + the on-device manual pass are the proof.
- 📥 **Android storage paths confirmed** — inboxed HUMAN item (step 6); needs physical hardware.

**Phase 03 status: COMPLETE (autonomous feature work).** All five feature ships landed on `main` (PRs #165, #167, #168, #172, #173), each through the full plan→apply→simplify→review→archive pipeline with an APPROVE gate, and the on-`main` Maestro E2E is green on both platforms. The codebase exit criteria — the add-a-calendar paths and the durable, migration-ready token identity — are met and verified in CI. The only remaining items are **irreducibly-human on-device/hardware passes** (the per-feature DoD device axes, on-disk restart durability, and the Android storage-path confirmation), all inboxed — none are blockers per the loop/DoD inbox convention.

## Risks & decisions

- Token persistence correctness is **critical** — it's the user's identity. Test restart/kill/reinstall-of-cache scenarios.
- Designed brand surface (onboarding) — needs the design artifacts, not just native defaults.

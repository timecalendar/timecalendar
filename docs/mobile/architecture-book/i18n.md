# i18n

> R-1 pointer style: where a rule is encoded (lint / type / CI gate), the prose
> points at the enforcing rule rather than restating it. The lint half —
> `i18next/no-literal-string` (no hardcoded user-facing strings) — lives in the
> Architecture Book "Lint & format"; this file owns the runtime half.

## Runtime — i18next + react-i18next (+ expo-localization)

- One module-scoped i18next instance lives in `src/i18n/index.ts`, created with
  `createInstance()` and `.use(initReactI18next)` so `useTranslation()`/`t()`
  resolve it **without an `<I18nextProvider>`** (initReactI18next registers it as
  react-i18next's default).
- Initialization is **synchronous** from bundled JSON catalogs — no network, no
  suspense/loading gate; the app renders already-localized. Wired by a side-effect
  `import "@/i18n"` at the top of `src/app/_layout.tsx`.
- `i18next` is the runtime because the live `i18next/no-literal-string` lint rule
  assumes i18next/`t()` idioms — any other library would orphan that rule.

## Locale detection — device locale, EN fallback, no switcher

- `src/i18n/detect-locale.ts` reads `expo-localization`'s device locales and returns
  the first that matches `fr` or `en`, else **`en`**.
- **EN (not FR) fallback is deliberate:** despite the French-first store identity, a
  predictable English baseline beats forcing French on the non-FR/EN long tail (e.g.
  a German device). The store app still ships FR for FR devices.
- `expo-localization` is a native module: declared in `app.config.ts` `plugins`;
  it autolinks and native projects are CNG-regenerated on the next `prebuild`.
- The in-app language switcher / persisted override **is wired by Settings** — see
  the Architecture Book "Settings preferences" (`src/features/settings/`): the
  startup `lng` resolves a stored preference before falling back to device
  detection, and the runtime setter calls `changeLanguage`.

## Flat, greppable keys — `keySeparator: false` + `nsSeparator: false`

- A key is one literal dotted string (`profile.tab.label`) stored verbatim as a flat
  top-level JSON key, so the string in code is byte-for-byte the string in the
  catalog — `grep` finds both call site and definition.
- Both separators are **disabled** so object-nesting is structurally unresolvable and
  the convention can't drift back to nested lookups; the dotted segments are a
  **naming convention**, not i18next structure.
- Plurals still work (i18next suffixes the flat key: `…_one` / `…_other`).
- **Escape hatch** if a catalog grows: split source JSON into multiple files merged
  into the one `translation` namespace at init. Reintroducing real `:` namespaces is
  the one thing that breaks the grep property, so we don't.

## Typed keys + FR/EN parity — both compile-time

- `src/i18n/i18next.d.ts` augments `react-i18next`'s `CustomTypeOptions` with
  `resources: { translation: typeof en }` and `keySeparator: false`, so the `t()` key
  argument is the union of EN catalog keys — a missing/mistyped key is a **`tsc`
  error**, not a silent key-as-fallback. EN (`src/i18n/locales/en.json`) is the
  canonical key set; FR (`fr.json`) is the translation.
- FR/EN parity is enforced **bidirectionally** in `index.ts` by typing each catalog
  against the other's key set (`Record<keyof typeof en, string>` and vice-versa): a
  *missing* FR key and an *extra* FR key both fail `tsc`. FR + EN are both required
  complete — a missing FR key fails CI.

## Proof in CI

- `src/i18n/i18n.test.tsx` renders a localized component through the real instance
  under the default locale and asserts the **translated** value renders (not the raw
  key) — proving init + `t()` + catalog resolve end to end, not merely that the files
  exist. i18n is initialized for the whole Jest suite via `jest/setup-i18n.ts`
  (`setupFilesAfterEnv`), mirroring the app's startup wiring, so every component test
  resolves real strings. Gated by the `test-mobile` job (tsc + lint + Jest), R-1.

## Deferred (recorded debt — not built)

- **Locale-aware date/time formatting — LANDED** (Phase-04). It is **not** an i18next
  concern: the `date-fns` display-only seam (`calendar/data/format.ts`, locale-resolved over a
  `LOCALES` map) covers every date/time across calendar/agenda/details/home — `formatFullDay`
  (the home today header, Phase-04 item 4) was the last addition, **closing roadmap item 5
  (date/time)**. Numeric date/time *values* come from that formatter, never from catalog keys
  (only counts/labels are catalog strings — e.g. the home `home.header.count` plural). **Relative
  time** ("in 30 min") + **ICU MessageFormat** remain the earned-when-needed debt below.
- **No relative-time formatting** ("in 30 min") — not needed yet (home shows absolute times, like
  Flutter); earns its place with the first feature that needs it.
- **No ICU MessageFormat, no catalog-parity/sort CI lint** — `tsc` covers parity
  today; revisit if catalogs grow.

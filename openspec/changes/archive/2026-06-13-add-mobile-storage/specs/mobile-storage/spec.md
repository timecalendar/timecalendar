## ADDED Requirements

### Requirement: Key-value storage behind a single wrapper seam
The app SHALL provide synchronous key-value persistence through a single `src/storage/`
module that exposes the app's own typed helpers over `react-native-mmkv`, rather than calling
`react-native-mmkv` directly at feature call sites, so the backend is swappable behind one
seam. The seam SHALL support storing and retrieving string, boolean, and number values, and
SHALL support checking existence and removing a key.

#### Scenario: Feature code does not import the KV backend directly
- **WHEN** a screen or feature needs to read or write a persisted key-value
- **THEN** it imports a helper from `@/storage`
- **AND** it does not import `react-native-mmkv` directly

#### Scenario: A value round-trips through the seam
- **WHEN** a value is stored under a key through the seam (string, boolean, or number)
- **THEN** reading that key through the seam returns the same value
- **AND** after the key is removed through the seam, existence-check for that key is false

### Requirement: Relational storage behind a single wrapper seam
The app SHALL provide a local relational (SQLite) store through a single `src/db/` module that
owns the `expo-sqlite` database handle and the Drizzle ORM instance, rather than opening the
database or constructing Drizzle at feature call sites. Feature code SHALL access the database
through `@/db` and SHALL NOT import `expo-sqlite` or `drizzle-orm` directly.

#### Scenario: One owned database handle
- **WHEN** the relational store is used
- **THEN** a single `expo-sqlite` handle and Drizzle instance from `src/db/` are used
- **AND** feature code does not import `expo-sqlite` or `drizzle-orm` directly

### Requirement: Database migrations applied at startup by a migration runner
The app SHALL apply an ordered, committed migration bundle to the local database at startup
via a migration runner built on Drizzle's `expo-sqlite` migrator. The runner SHALL be
idempotent (running it when no new migrations exist is a no-op that leaves the database
unchanged) and SHALL record migration failures through the `@/firebase` observability seam.
The committed migration bundle MAY be empty (this foundation step ships zero feature
migrations); feature schemas and their migrations are added by the features that own them.

#### Scenario: Runner runs at startup
- **WHEN** the app launches
- **THEN** the migration runner applies the committed migration bundle to the database
- **AND** Drizzle's migration tracking table is created/advanced accordingly

#### Scenario: Empty bundle is a valid green run
- **WHEN** the committed migration bundle contains no migrations
- **THEN** the runner completes successfully, leaving the database at its initial version

#### Scenario: Migration failure is observable
- **WHEN** applying a migration fails at startup
- **THEN** the error is recorded through the `@/firebase` seam

### Requirement: Storage backends are native modules configured through CNG
The storage native modules SHALL be configured through `app.config.ts` and Expo config
plugins so native projects are regenerated by `expo prebuild`, never hand-edited. `expo-sqlite`
SHALL be registered as a config plugin; `react-native-mmkv` (v4 / Nitro) autolinks without a
plugin entry.

#### Scenario: Native config flows through prebuild
- **WHEN** the native projects are regenerated with `expo prebuild`
- **THEN** the SQLite native module is configured via the `expo-sqlite` config plugin in `app.config.ts`
- **AND** no native iOS/Android files are hand-edited to enable storage

### Requirement: SQL migration bundling toolchain is wired
The project SHALL include the toolchain Drizzle requires to bundle SQL migration files for
Expo: a `drizzle.config.ts` (SQLite dialect, Expo driver), a migration-generation npm script,
the Babel `inline-import` plugin for `.sql` files, and the Metro `.sql` source extension — so
that a feature adding a schema generates its migration with a single command rather than
introducing build plumbing.

#### Scenario: A feature can generate a migration without adding plumbing
- **WHEN** a feature adds a Drizzle schema and runs the migration-generation script
- **THEN** the generated `.sql` migration is bundled into the committed migration set via the
  already-configured Babel/Metro toolchain
- **AND** no Babel, Metro, or Drizzle configuration needs to be added for that feature

### Requirement: Storage backends are accessed only through their seams (lint-enforced)
The codebase SHALL forbid importing the storage backends (`react-native-mmkv`, `expo-sqlite`,
`drizzle-orm` and its subpaths) anywhere except the seam directories (`src/storage/**`,
`src/db/**`), enforced by a lint rule, so the swappable-seam boundary cannot be bypassed
silently.

#### Scenario: Direct backend import outside the seam fails lint
- **WHEN** a file outside `src/storage/` or `src/db/` imports `react-native-mmkv`,
  `expo-sqlite`, or `drizzle-orm`
- **THEN** `npm run lint` reports an error

### Requirement: Storage wiring is verified by automated tests
The unit test suite SHALL include a test that round-trips a value through the `@/storage`
seam, and a test that drives the migration runner and asserts it invokes Drizzle's migrator
with the committed bundle, so the storage wiring is proven in CI. (Confirming that database
tables actually materialize on disk is an explicitly on-device step — the Maestro e2e
exercises the real runner by launching the app — not a CI unit-test gate, because `expo-sqlite`
has no off-device JavaScript implementation.)

#### Scenario: KV seam round-trip proven in CI
- **WHEN** the proof test sets and reads values through the `@/storage` seam
- **THEN** the read values equal the written values

#### Scenario: Migration runner wiring proven in CI
- **WHEN** the proof test invokes the migration runner with the SQLite seam mocked
- **THEN** Drizzle's `migrate()` is called with the committed migration bundle

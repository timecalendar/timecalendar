## ADDED Requirements

### Requirement: Personal events data layer lives in a feature folder over the database seam
The Personal events feature's device-local data layer SHALL live in a feature folder
`mobile/src/features/personal-events/`, with the repository, domain types, row↔domain mappers,
uid generator, and reactive read hook under a `data/` sub-layer, mirroring the feature-module
shape Settings established. The Drizzle schema SHALL live at `mobile/src/db/schema.ts` (the
database seam dir). This change SHALL add no Personal events screen, route, form, native picker,
or list rendering (those are a separate change).

#### Scenario: Data layer lives under the feature folder, schema under the db seam
- **WHEN** the repository, domain types, mappers, uid generator, or read hook are located
- **THEN** they are under `mobile/src/features/personal-events/data/`
- **AND** the Drizzle schema is at `mobile/src/db/schema.ts`
- **AND** no `mobile/src/app/` route or screen is added for Personal events by this change

#### Scenario: Feature code does not import the database backend directly
- **WHEN** the repository or read hook reads or writes personal events
- **THEN** it imports `{ db }`, the schema, and any query helpers from `@/db`
- **AND** it does not import `drizzle-orm` or `expo-sqlite` directly

### Requirement: Personal events schema mirrors the Flutter wire format for importer fidelity
The app SHALL define a `personalEvents` Drizzle table (SQL `personal_events`) whose columns
mirror the Flutter `PersonalEvent.toMap()` wire format so the Phase-09 importer can write
recovered rows with no data loss and minimal transformation: a `uid` TEXT primary key, a `title`
TEXT not-null, a `color` TEXT not-null holding the `#RRGGBB` hex string verbatim, `startsAt`,
`endsAt`, and `exportedAt` TEXT not-null holding UTC ISO-8601 strings, and nullable `location`
and `description` TEXT columns. The constant `kind` (`EventKind.Personal`) SHALL NOT be stored.

#### Scenario: The schema columns match the importer wire format
- **WHEN** the `personalEvents` table is defined
- **THEN** it has the columns `uid` (TEXT primary key), `title`, `color`, `startsAt`, `endsAt`,
  `exportedAt` (TEXT not-null), and `location`, `description` (TEXT nullable)
- **AND** date columns hold UTC ISO-8601 strings and `color` holds the `#RRGGBB` hex string

#### Scenario: A `toMap()`-shaped record imports without transformation
- **WHEN** a record in the Flutter `toMap()` wire format (uid, hex color, UTC ISO-8601 date
  strings, nullable location/description) is written into the schema
- **THEN** every field is stored verbatim with no parse-and-re-encode and no data loss

### Requirement: First real migration generated and committed
The change SHALL replace the empty migration bundle with the first real migration, generated by
`drizzle-kit generate` (driver `expo`) from `src/db/schema.ts`: a `0000_*.sql` creating the
`personal_events` table, an advanced `meta/_journal.json`, and an updated `migrations.js`, all
committed. The startup migration runner (`runMigrations()`) SHALL apply this migration unchanged
in its wiring.

#### Scenario: The committed bundle contains the personal_events migration
- **WHEN** the migration bundle is generated and committed
- **THEN** `src/db/migrations` contains a `CREATE TABLE` migration for `personal_events` and a
  journal entry referencing it
- **AND** the bundle is committed (a fresh clone has the migration without running codegen)

#### Scenario: The runner drives the non-empty bundle
- **WHEN** the app starts and `runMigrations()` runs
- **THEN** it drives `migrate(db, migrations)` with the committed bundle
- **AND** the existing migration-runner proof test still passes against the now non-empty bundle

### Requirement: Repository CRUD and date-range query over the database seam
The Personal events feature SHALL expose an async repository over the `@/db` seam with: a
`findAll` returning all events, a `getById(uid)` returning one event or undefined, an `upsert`
that inserts-or-updates by `uid` (mirroring the Flutter `put`), a `remove(uid)` that deletes by
`uid`, and a `findInRange` that returns events within a date range (for the eventual
calendar/list). The repository SHALL map stored rows to a domain type and domain objects to rows
through pure mappers, and SHALL NOT import the database backend directly.

#### Scenario: Upsert inserts a new event and updates an existing one
- **WHEN** `upsert` is called with an event whose `uid` does not yet exist
- **THEN** a new row is inserted
- **WHEN** `upsert` is called again with the same `uid` and changed fields
- **THEN** the existing row is updated (no duplicate row)

#### Scenario: getById and remove operate by uid
- **WHEN** `getById(uid)` is called for a stored event
- **THEN** it returns that event mapped to the domain type
- **WHEN** `remove(uid)` is called
- **THEN** the row for that `uid` is deleted and a subsequent `getById(uid)` returns undefined

#### Scenario: findInRange returns events in chronological range order
- **WHEN** `findInRange(from, to)` is called
- **THEN** it returns the events overlapping the range, correctly ordered chronologically by
  start time (the UTC ISO-8601 text columns sort lexicographically as chronological)

### Requirement: Row↔domain mappers normalize dates to canonical UTC
The data layer SHALL convert between the stored row type (TEXT ISO-8601 date strings) and a
domain type exposing `Date` timestamps through pure mappers. Writing a domain object SHALL
normalize each timestamp to a canonical UTC ISO-8601 string (so the text columns' lexicographic
order is chronological), and reading SHALL parse the stored string back to a `Date`. Nullable
`location`/`description` SHALL round-trip including absent values.

#### Scenario: A domain object round-trips through the mappers
- **WHEN** a domain event is mapped to a row and back to a domain event
- **THEN** all fields are preserved, the timestamps are equal, and the row's date strings are
  canonical UTC ISO-8601
- **AND** a null `location` or `description` round-trips as absent

### Requirement: New event ids are generated by a swappable uid wrapper
New locally-created events SHALL receive a UUID from a single uid generator wrapper (over
`expo-crypto`'s `randomUUID`), and the repository SHALL also accept a caller-supplied uid (so the
importer can supply a recovered uid). The wrapper SHALL be the only import site of the underlying
generator.

#### Scenario: A new event gets a generated uid
- **WHEN** the uid wrapper is called for a new event
- **THEN** it returns a UUID string

#### Scenario: The importer supplies its own uid
- **WHEN** the repository's upsert is given an event carrying an existing uid
- **THEN** it uses that uid as the row identity (no new uid is generated)

### Requirement: Reactive read hook over the database seam
The feature SHALL expose a reactive read hook that returns the current list of personal events
and re-renders consumers when the underlying table changes, built on the seam's reactive
`useLiveQuery` (so feature code does not import the database backend directly).

#### Scenario: The reactive hook reflects the current rows
- **WHEN** a component uses the personal-events read hook
- **THEN** it receives the current events mapped to the domain type
- **AND** the hook is built on `useLiveQuery` re-exported from `@/db`, not a direct
  `drizzle-orm` import

### Requirement: The data layer is verified by automated tests under the coverage gate
The unit test suite SHALL prove the data layer: the row↔domain mappers (round-trip + canonical
UTC + null handling), the uid wrapper, and the repository CRUD + range query (proven against the
`@/db` seam). These tests SHALL run under the existing `test-mobile` CI job, and the logic paths
(`src/db/**`, `src/features/**`) SHALL meet the enforced K-3 coverage threshold (90% lines and
branches) on landing.

#### Scenario: The data layer meets the coverage threshold
- **WHEN** the change's test suite is run with coverage
- **THEN** all configured thresholds pass (the new `src/db/**` and `src/features/**` paths at
  90% lines and branches)

#### Scenario: The repository is proven against the seam in CI
- **WHEN** the repository proof test drives upsert/getById/remove/findInRange with the `@/db`
  seam mocked
- **THEN** each operation issues the expected query and maps results to the domain type

# Design — Personal events data layer (first Drizzle schema + migration + repository, importer-targetable)

## Context

This is Phase 2's **Feature B, data layer only** (ADR 004 — Settings → **Personal events** →
School selection). Settings stressed flat KV preferences over `@/storage`; Personal events
stresses the next axis up: **structured device-local relational data** — a real Drizzle schema,
the first real migration, a repository/query layer over the `@/db` seam. The forms, native
date/time pickers, list rendering, and route are a later B-issue and are out of scope.

The foundation **Storage** step (`add-mobile-storage`) built the `@/db` seam, the startup
`runMigrations()` runner, the SQL-bundling toolchain (`drizzle.config.ts`, `babel.config.js`
inline-import, `metro.config.js` `.sql` sourceExt), the seam-import lint boundary, and a
committed-but-**empty** migration bundle — explicitly so "the first feature regenerates the
bundle via `npm run generate:migrations` once it adds `src/db/schema.ts`" and "the first feature
owns the first migration and the first key." This change discharges all of that.

Two hard constraints shape every decision:

- **Importer fidelity (load-bearing).** The Phase-09 one-shot importer replays the Flutter
  app's sembast JSONL and writes the recovered `personal_events` into this schema. Its wire
  format is `PersonalEvent.toMap()` in `app/lib/modules/personal_event/models/personal_event.dart`:
  `uid` (UUID string, the sembast record key), `title` (String), `color` (`ColorUtils.colorToHex`
  → a `#RRGGBB` 6-digit hex string — the alpha byte is stripped by `substring(2)`), `startsAt` /
  `endsAt` / `exportedAt` (`DateTime.toUtc().toIso8601String()` → UTC ISO-8601 strings),
  `location` / `description` (nullable String). `kind` (`EventKind.Personal`) is a constant, not
  stored. The schema must let the importer write rows with **no data loss and minimal
  transformation**.
- **Seam discipline (lint-enforced).** `drizzle-orm` and `expo-sqlite` are banned outside
  `src/db/**` / `src/storage/**` (`storageBackendImportPatterns` in `eslint.config.js`).
  `sqliteTable` / column builders import `drizzle-orm/sqlite-core`, so `schema.ts` **must** live
  at `src/db/schema.ts`. `useLiveQuery` imports `drizzle-orm/expo-sqlite`, so the reactive read
  hook **must** be built in the seam and re-exported, OR the feature imports only `{ db }` and
  the table from `@/db` and calls `useLiveQuery` from the seam — see D5. The feature repository
  imports `{ db }` from `@/db` and the table from `@/db/schema` — lint-clean.

## Goals / Non-Goals

**Goals:**
- A first real Drizzle schema (`personalEvents`) whose columns mirror the Flutter `toMap()` wire
  format so the Phase-09 importer targets it with no data loss.
- The first real migration bundle, regenerated by `drizzle-kit generate` (driver `expo`),
  committed; the existing `runMigrations()` runner now applies one migration instead of zero
  (runner + `migrate.test.ts` unchanged).
- A repository CRUD + date-range query over the `@/db` seam, the row↔domain mappers, a uid
  generator, and a reactive `useLiveQuery` read hook — the data/logic layer the future B-screen
  consumes.
- The K-3 coverage gate (already enforced) stays green: `src/db/**` and `src/features/**` meet
  90% lines+branches.

**Non-Goals:**
- **No Personal events screen, route, native date/time pickers, forms, or list rendering** — a
  separate B-issue. No `src/app/` change.
- **No calendar overlay / timeline** — Phase 05, behind a clean seam (the roadmap is explicit).
- **No server sync / TanStack Query** — personal events are device-local; there is no server
  endpoint for them (the roadmap calls them "irreplaceable on-device data, no server backup").
- **No `useMigrations()` blocking gate conversion** — see D6; this feature's data is not read
  before first paint, so the splash/`useAppReady` gate is unchanged.
- **No MMKV encryption / secure-store** — personal events are non-sensitive user content;
  encryption stays the storage change's recorded debt.
- **No `eslint-plugin-boundaries`** — the feature-boundary lint is still TIM-135's; this change
  establishes the second feature folder it will govern.
- **No generic ORM/repository framework** — one feature's typed repository, not a base class
  (R-2 — the golden-path exemplar is earned in Phase 1.5 from three features, not declared).

## Decisions

### D1 — Schema: `personalEvents` table mirrors the Flutter `toMap()` wire format
`src/db/schema.ts` defines one table, `personalEvents` (SQL table name `personal_events`,
matching the Flutter sembast store name so the importer maps store→table by name):

| Drizzle column | SQL | Source (`toMap()`) | Notes |
| --- | --- | --- | --- |
| `uid` | `text` PRIMARY KEY | `uid` | the sembast record key (a UUID); upsert key |
| `title` | `text` NOT NULL | `title` | |
| `color` | `text` NOT NULL | `color` | the `#RRGGBB` hex string, stored verbatim (D3) |
| `startsAt` | `text` NOT NULL | `startsAt` | UTC ISO-8601 string (D4) |
| `endsAt` | `text` NOT NULL | `endsAt` | UTC ISO-8601 string (D4) |
| `exportedAt` | `text` NOT NULL | `exportedAt` | UTC ISO-8601 string (D4) |
| `location` | `text` (nullable) | `location` | nullable String? |
| `description` | `text` (nullable) | `description` | nullable String? |

`kind` is not stored (a constant, `EventKind.Personal`). The Drizzle TS column names are
camelCase (`startsAt`), the SQL names are the snake/verbatim names the importer's `toMap()` keys
use where it matters — but because the importer writes through *this* schema (not raw SQL), the
binding is the column→domain mapping, and the importer maps its `toMap()` keys to these columns
once. `uid` is the explicit primary key (`.primaryKey()`), not an auto-increment surrogate,
because the upsert-by-uid semantics (Flutter's `_store.record(uid).put`) and the importer's
record key both require the uid to be the identity.
*Alternative:* a surrogate integer `id` + a unique `uid` (rejected — the uid *is* the identity in
both the Flutter store and the importer wire format; a surrogate adds a column with no consumer
and complicates upsert — R-2).

### D2 — uid generation: `expo-crypto`'s `randomUUID()`, wrapped (caller may supply)
New locally-created events need a UUID; the importer supplies its own (the recovered uid). The
repository's `upsert`/`create` accepts the uid, and a thin `newEventId()` wrapper in
`src/features/personal-events/data/uid.ts` produces one for new events via **`expo-crypto`'s
`randomUUID()`** (a v4 UUID, RFC-4122 — the same shape the Flutter `uuid` package emits, so
local and imported uids are indistinguishable and never collide). `expo-crypto` is added because
RN/Hermes has no global `crypto.randomUUID()` and `expo-crypto` is the Expo-lane, autolinking
choice (vs. pulling a pure-JS `uuid` package — `expo-crypto` uses the platform CSPRNG, stays in
the Expo upgrade lane, and is the documented Expo way).
**Native-affecting (flagged):** `expo-crypto` is a native module that **autolinks**; it requires a
`npx expo prebuild` to regenerate the native projects (CNG; the e2e build already prebuilds).
**Implementation note (deviation):** `expo-crypto` ships **no config plugin** (`app.plugin.js`),
so it is **not** added to `app.config.ts` `plugins` — adding it there makes `expo prebuild` throw.
It autolinks with no plugin entry (like `react-native-mmkv`); `package.json` + autolink is the
wiring, verified by a clean `prebuild`. Under Jest the wrapper is mocked (no native module), so
`tsc`/lint/Jest are unaffected. The wrapper keeps the generator swappable (one import site) and
lets the importer bypass it.
*Alternative:* a pure-JS `uuid` dep (rejected — adds a JS dep that duplicates platform crypto and
isn't in the Expo lane); a caller-only uid with no generator (rejected — the future create-form
needs a generator, and centralizing it now is the seam, not speculation, since `create` is a
named repository operation).

### D3 — Color stored as the `#RRGGBB` hex string, verbatim (importer fidelity)
The Flutter `color` is serialized by `ColorUtils.colorToHex` to a `#`-prefixed 6-digit hex
string (alpha stripped). Stored **verbatim as TEXT**. The importer writes the `toMap()` `color`
value with zero transformation; the future UI parses `#RRGGBB` → a render color at the
presentation edge (not the data layer's concern). Storing an integer ARGB (or a normalized color
token) would force the importer to parse/re-encode and risk an alpha mismatch (Flutter strips
alpha on write; re-adding it on import is a transformation that can drift). TEXT-verbatim is the
zero-loss, zero-transformation choice.
*Alternative:* `integer` packed RGB (rejected — the importer would have to parse the hex and the
UI would have to re-format; verbatim text is lossless and trivially round-trips); a constrained
palette enum (rejected — the Flutter app allows arbitrary colors; constraining loses data).

### D4 — Dates stored as TEXT ISO-8601 UTC strings (ADR 011 — the load-bearing decision)
**The choice:** `startsAt` / `endsAt` / `exportedAt` are **TEXT, holding UTC ISO-8601 strings**
(e.g. `2026-06-14T09:00:00.000Z`) — *not* `integer({ mode: "timestamp_ms" })` epoch-ms. This
constrains the importer and every future query, so it earns **ADR 011**.

Weighed against both axes the issue demands:

- **(a) Importer fidelity.** The Flutter `toMap()` emits exactly UTC ISO-8601 strings
  (`startsAt.toUtc().toIso8601String()`). TEXT-verbatim means the importer writes the string with
  **no parse, no transformation, no precision loss, no timezone ambiguity** — the highest-fidelity
  path, and fidelity is the *non-negotiable* of the irreplaceable-data importer. An epoch-ms
  column would force the importer to `Date.parse()` each ISO string and risk (i) sub-ms / format
  edge cases, (ii) a silent failure on any non-canonical string becoming `NaN` → a corrupt row in
  a one-shot, no-server-backup recovery. The cost of a bad import is unrecoverable user data;
  fidelity wins.
- **(b) Range-query needs (the eventual calendar/list).** A `findInRange(fromIso, toIso)` over
  `startsAt`/`endsAt` is a `BETWEEN` / `>=`/`<=` comparison. **Lexicographic ordering of
  canonical UTC ISO-8601 strings is identical to chronological ordering** (fixed-width,
  zero-padded, `Z` suffix) — so a TEXT column sorts and range-filters correctly with a plain
  index, no conversion. The query property the calendar needs (ordered range scan) holds on TEXT
  for canonical UTC ISO-8601. The one caveat — it only holds for *canonical* UTC strings — is
  *guaranteed* here because every write goes through the repository's mappers (D7) which
  normalize to `toISOString()` (always canonical UTC), and the importer's source is already
  canonical UTC from Flutter.

**The tradeoff stated plainly:** epoch-ms is the more "native-numeric" representation and is
marginally cheaper to compare numerically, but it costs importer fidelity (a lossy parse on a
one-shot recovery) and buys nothing the TEXT range query can't do for our canonical-UTC data. We
trade a hair of numeric-comparison tidiness for **lossless one-shot recovery + a correct,
indexed range query** — the right call for irreplaceable data. The domain type exposes `Date`
objects (D7 mappers convert at the edge), so consumers never touch the string format.
*Alternatives recorded in ADR 011:* `integer({ mode: "timestamp_ms" })` (rejected — lossy
importer parse, the fidelity risk above); store a `Date` via Drizzle's `timestamp` mode (the same
epoch-int underneath — same rejection); store the local-time string (rejected — Flutter stores
UTC; local-time loses the offset and breaks cross-timezone correctness).

### D5 — Repository over `@/db`; reactive read via `useLiveQuery` through the seam
The repository (`src/features/personal-events/data/repository.ts`) is a set of **pure async CRUD
functions** over the `@/db` seam — no class, no base repository (R-2, mirroring Settings' module
of functions):
- `findAll(): Promise<PersonalEvent[]>` — `db.select().from(personalEvents)`, mapped to domain.
- `getById(uid): Promise<PersonalEvent | undefined>` — `…where(eq(uid))`, `.get()` / first row.
- `upsert(event): Promise<void>` — `db.insert(personalEvents).values(row).onConflictDoUpdate({
  target: personalEvents.uid, set: row })` — mirrors Flutter's `put` (upsert by uid); the single
  write path for both new (create) and edit (update).
- `remove(uid): Promise<void>` — `db.delete(personalEvents).where(eq(uid))` — mirrors `delete`.
- `findInRange(fromIso, toIso): Promise<PersonalEvent[]>` — the date-range query the future
  calendar/list needs (events overlapping the closed window `[from, to]`); justifies the D4 TEXT-range property.
  Earned now because it's the query the schema's date choice exists to serve, and proving it now
  validates D4; the UI that calls it is the later B-issue.

`eq` / `and` / `gte` / `lte` import from `drizzle-orm` — **banned outside the seam**. So either
(i) the repository lives partly in the seam, or (ii) the seam re-exports the query helpers the
feature needs. **Chosen:** the feature repository imports `{ db }` and the table from `@/db`, and
the **query operators it needs (`eq`, `and`, `gte`, `lte`) are re-exported from `@/db`** (a small,
explicit seam surface — the encoded form of "the feature never imports `drizzle-orm`"). This keeps
the repository in the feature folder (its domain) while honoring the lint boundary, exactly as
Settings' reactive `useStoredString` lives in `@/storage`. The reactive read hook
(`usePersonalEvents`) needs `useLiveQuery` (imports `drizzle-orm/expo-sqlite`) — also re-exported
from `@/db` as `useLiveQuery` (the storage section already names `useLiveQuery` as the pattern
features inherit; this wires it into the seam). The feature `hooks.ts` calls the seam's
`useLiveQuery(db.select().from(personalEvents))` and maps rows→domain.
*Alternative:* put the whole repository in `src/db/` (rejected — the repository is the *feature's*
domain logic, not the backend seam; conflating them blocks the feature-boundary lint later and
breaks the Settings-established convention that the feature owns its `data/` layer); widen the
seam to re-export all of `drizzle-orm` (rejected — re-export only what a consumer needs, R-2).

### D6 — No `useMigrations()` blocking gate this change (the data isn't read before paint)
The storage section names `useMigrations()` as the loading-gate pattern "when a feature's first
read must wait on its tables," and `use-app-ready.ts` is the documented adoption point. **This
change does not adopt it.** Personal events are read by the future B-screen (and the Phase-05
calendar), not at first paint — the splash gate has no dependency on the `personal_events` table
existing. The first real migration is still **idempotent and fast** (one `CREATE TABLE`),
applied fire-and-forget by the existing `runMigrations()` at startup; a failure already reaches
`@/firebase` (unchanged). Converting `useAppReady` to block on migrations now would be
speculative (R-2) — it's earned by the first feature whose *initial read must block on a table*,
which is not this data-layer change. Recorded so the future B-screen / calendar inherits the open
question.
*Alternative:* adopt `useMigrations()` now "to be safe" (rejected — no screen reads the table at
startup; blocking first paint on a migration nothing-yet-depends-on is the speculative gate R-2
forbids).

### D7 — Row↔domain mappers isolate the storage format from the domain type
The domain type `PersonalEvent` (`data/types.ts`) exposes ergonomic types — `Date` for the three
timestamps, `string` for `color`/`title`/`location`/`description` — while the row type (Drizzle's
inferred `typeof personalEvents.$inferSelect` / `$inferInsert`) holds the TEXT ISO strings.
**`rowToEvent(row)` and `eventToRow(event)`** are pure functions converting at the boundary:
`new Date(row.startsAt)` ↔ `event.startsAt.toISOString()` (always canonical UTC — the property
D4's range query relies on), nullable `location`/`description` passed through. These mappers are
the **pure-logic core** that carries most of the 90% coverage and the importer-fidelity guarantee
(the importer can reuse `eventToRow` after building a domain object, or write the `toMap()`
strings directly — both land canonical rows). Keeping mappers pure (no `db`) lets them be unit-
tested exhaustively without any SQLite mock.
*Alternative:* expose the raw row type as the domain type (rejected — leaks the TEXT storage
format to every consumer and forces each to parse dates; the mapper is one place, tested once).

### D8 — Testing strategy: pure mappers/uid unit-tested; repository proven at the `@/db` seam
The K-3 logic globs (`src/db/**`, `src/features/**`) demand 90% lines+branches. Strategy mirrors
the storage harness's "mock at the seam, exercise everything above it":
- **Mappers + uid wrapper** (pure, no I/O) — unit-tested directly and exhaustively (round-trip
  every field, null handling, canonical-UTC normalization, the uid wrapper delegates to the mocked
  `expo-crypto`). This is where most logic coverage lands.
- **Repository** — `expo-sqlite` / `drizzle-orm/expo-sqlite` have **no off-device JS** (the same
  situation as firebase's native modules), so a real on-disk round-trip isn't available under
  Jest. The repository is proven by **mocking the `@/db` `db` object** (a query-builder spy) and
  asserting each function issues the expected Drizzle chain (`insert…onConflictDoUpdate` for
  upsert, `delete…where` for remove, `select…where` for getById, the `gte`/`lte` range for
  `findInRange`) and maps the returned rows→domain via the (real) mappers. This proves the
  repository **wiring** (the query shape + the mapping) in CI; the real table materialization is
  on-device — the Maestro e2e launches the app, `runMigrations()` creates the table, and the
  future B-screen's flow exercises real rows (the same CI-proves-wiring / on-device-proves-
  materialization split the storage section established).
- **`migrate.test.ts` is unchanged** — it asserts `runMigrations()` drives `migrate(db,
  migrations)` with the committed bundle; the bundle now has one entry, the assertion still holds.
- **`setup-db.ts` may need to expose a richer `db` mock** so repository tests can drive the query
  chain (today it mocks `drizzle` → `{}`). The implementer extends the per-test mock (or a local
  `jest.mock("@/db")`) so the repository's chain is spy-able — recorded as a task; keep the
  suite-wide `setup-db` minimal and do the query-builder spying per-test.

If a configured logic path can't reach 90% after these, **prefer adding a test**; only exclude a
genuinely-untestable path from the glob **with a recorded reason** in `jest.config.js` (the DoD's
no-third-state rule) — but the mappers/uid/repository are all testable, so no exclusion is
expected.

## Risks / Trade-offs

- **Importer fidelity is the whole point and is one-shot (highest stakes).** D3/D4 choose
  verbatim TEXT precisely so the importer writes recovered data with no transformation. The risk
  is a schema choice that forces a lossy import; mitigated by storing exactly the `toMap()` wire
  format and by ADR 011 pinning the decision + its revisit. The mappers' canonical-UTC
  normalization is tested.
- **TEXT range query depends on canonical UTC strings.** Lexicographic = chronological only for
  fixed-width canonical UTC ISO-8601. Mitigated: every write normalizes via `toISOString()` (D7),
  the importer's source is already canonical UTC, and `findInRange`'s test asserts ordering on
  representative timestamps. Recorded as the one caveat in ADR 011.
- **`expo-crypto` is a new native dep (prebuild required).** Mitigated: it autolinks, the e2e
  build already prebuilds, and the uid wrapper is mocked under Jest so CI `test-mobile` is
  unaffected. Flagged in tasks as a `prebuild` step (config-shape, verified by prebuild/e2e, not
  lint — R-1).
- **Coverage gate must stay green.** `src/db/schema.ts` is declarative (Drizzle table) — mostly
  covered by being imported; the logic (mappers, uid, repository) carries the bar. Mitigated by
  D8's strategy + the explicit "verify `npm test -- --coverage` passes locally" task. The
  fallback is to test, not to exclude.
- **Repository proven by query-shape assertion, not a real DB.** This proves wiring, not that
  SQLite accepts the SQL. Mitigated: `drizzle-kit generate` validates the schema→SQL at generation
  time, `runMigrations()` runs the real `CREATE TABLE` on-device (Maestro launch), and the schema
  is a single plain table. Accepted, mirroring the storage section's CI-vs-on-device split.

## Migration Plan

Additive; rollback = revert (the new table is created by the first migration — a revert removes
the schema/migration, and on a dev/e2e device the `personal_events` table simply goes unused;
no production data exists yet). Order: write `src/db/schema.ts` → run
`npm run generate:migrations` and commit the real bundle → confirm `migrate.test.ts` still passes
→ add `expo-crypto` (dep + `app.config.ts` plugin) → re-export `eq`/`and`/`gte`/`lte` +
`useLiveQuery` from `@/db` → build the feature `data/` layer (`types.ts` mappers → `uid.ts` →
`repository.ts` → `hooks.ts` → `index.ts`) + tests → extend the Jest `@/db`/setup-db mock for
repository query-shape assertions → **verify `npm test -- --coverage` passes the gate** → docs +
ADR 011 + changelog + roadmap → `npx expo prebuild` (verify `expo-crypto` links) → gate on
`tsc` / lint / test+coverage. No OpenAPI regen, no server/web/`app/` change, no CI workflow
change, no inbox handoff.

## Why

Phase 2 ([TIM-3](/TIM/issues/TIM-3)) is the dependency-upgrade & maintenance
phase. The B1 audit ([TIM-36](/TIM/issues/TIM-36)) listed B7 ([TIM-49](/TIM/issues/TIM-49))
as a Wave 3 row covering `flutter_local_notifications` 19→21 plus four
plus-plugin / icon / lint majors. The TIM-95 Plan re-audit found
`flutter_local_notifications` is a **zombie dependency**: declared in
`app/pubspec.yaml` at `^19.2.1` but with 0 `package:flutter_local_notifications/`
imports across `app/lib/`, `app/test/`, and `app/integration_test/`. All
push notifications are wired through `firebase_messaging` (FCM `onMessage`
handler in `app/lib/modules/firebase/services/notification/notification.dart`);
no local scheduling exists. The 19→21 "changed init / permission / Android
scheduling APIs" risk evaporates — same shape as
[TIM-91](/TIM/issues/TIM-91)'s `google_sign_in` finding — and the right
action is to delete the dep instead of migrating it.

The other four B7 items are real bumps with active call sites and must
land together so Wave 3 ships against one regression gate:

- `device_info_plus` ^11.4.0 → ^13 — used by the suggestion module
  (`format_device_info.dart` + `suggestion_screen.dart`) via
  `BaseDeviceInfo.data`.
- `package_info_plus` ^8.3.0 → ^10 — used by
  `settings/providers/settings_provider.dart` and
  `test/support/settings_provider.dart` via `PackageInfo.fromPlatform()`.
- `font_awesome_flutter` ^10.5.0 → ^11 — 17 distinct icon constants in
  use across ~15 files; v11 may rename or drop some.
- `flutter_lints` ^5.0.0 → ^6 (dev) — re-run `flutter analyze` and
  address any new lints.

## What Changes

- **Remove the direct dependency** `flutter_local_notifications: ^19.2.1`
  from `app/pubspec.yaml`. `pubspec.lock` is regenerated by `flutter pub get`
  and drops `flutter_local_notifications`,
  `flutter_local_notifications_linux`,
  `flutter_local_notifications_platform_interface`, and
  `flutter_local_notifications_windows`.
- **Surgically remove the Pods-managed entries** for
  `flutter_local_notifications.framework` from the `[CP] Embed Pods
  Frameworks` script phase in `app/ios/Runner.xcodeproj/project.pbxproj`
  (2 lines: one `inputPaths` entry + one matching `outputPaths` entry).
  CocoaPods would normally rewrite these on a Mac `pod install`; on the
  Linux Applier host the edit is mechanical and idempotent.
- **Remove the stale `flutter_local_notifications` comment** in
  `app/android/app/build.gradle` (the "requires desugaring" two-line
  comment above `compileOptions`). Keep `coreLibraryDesugaringEnabled true`
  and the `desugar_jdk_libs` dependency line in place — desugaring is a
  conservative default and removing it is a separate decision.
- **Bump `device_info_plus`** `^11.4.0` → `^13` in `app/pubspec.yaml`.
  Verify `BaseDeviceInfo.data` access still resolves in
  `app/lib/modules/suggestion/utils/format_device_info.dart` and
  `app/lib/modules/suggestion/screens/suggestion_screen.dart`.
- **Bump `package_info_plus`** `^8.3.0` → `^10` in `app/pubspec.yaml`.
  Verify `PackageInfo.fromPlatform()`, `.version`, `.buildNumber`,
  `PackageInfo.setMockInitialValues(...)` still resolve in
  `app/lib/modules/settings/providers/settings_provider.dart` and
  `app/test/support/settings_provider.dart`.
- **Bump `font_awesome_flutter`** `^10.5.0` → `^11` in `app/pubspec.yaml`.
  Verify each of the 17 in-use constants (`calendar`, `calendarDays`,
  `eyeSlash`, `gear`, `graduationCap`, `house`, `info`, `list`,
  `magnifyingGlass`, `paperPlane`, `pencil`, `plus`, `squareCheck`,
  `toggleOn`, `upRightFromSquare`, `user`, `xmark`) still resolves in
  v11. If any constant was renamed, update its call site; if any was
  dropped, escalate to FoundingEngineer before guessing a replacement.
- **Bump `flutter_lints`** `^5.0.0` → `^6.0.0` in
  `dev_dependencies`. Re-run `flutter analyze` and address every new
  lint surfaced.

Out of scope: any local-notification feature work (the dep is being
deleted, not migrated to a different plugin), any `firebase_messaging`
refactor (separate suite, untouched), any Android Gradle min-SDK or
Java-compat bumps (`coreLibraryDesugaringEnabled` stays as a conservative
default; min-SDK changes belong to a future platform-minimum decision),
any backend / `openapi/` / `web/` / CI edits.

### Modified Capabilities
- `flutter-dependency-hygiene`: adds B7 requirements that
  `flutter_local_notifications` is removed as a zombie dep alongside its
  iOS native artifact cleanup, and that the four B7 plus-plugin / icon /

  B6 zombie-`google_sign_in` requirement added by
  [TIM-91](/TIM/issues/TIM-91).

## Impact

- `app/pubspec.yaml` —
  `flutter_local_notifications` removed from `dependencies:`;
  `device_info_plus`, `package_info_plus`, `font_awesome_flutter` bumped
  to their new major majors; `flutter_lints` (dev) bumped to ^6.
- `app/pubspec.lock` — regenerated by `flutter pub get`; four
  `flutter_local_notifications*` packages drop out and the four bumped
  packages move forward (with their own transitive shifts).
- `app/ios/Runner.xcodeproj/project.pbxproj` — 2 stale Pods-managed
  input/output path entries removed (the `flutter_local_notifications`
  framework pair) from the `[CP] Embed Pods Frameworks` script phase.
- `app/android/app/build.gradle` — the two-line
  `// flutter_local_notifications requires desugaring` comment block is
  removed; the `coreLibraryDesugaringEnabled true` / `desugar_jdk_libs`
  lines stay.
- `app/ios/Runner/GeneratedPluginRegistrant.m` and
  `app/android/app/src/main/java/io/flutter/plugins/GeneratedPluginRegistrant.java`
  — regenerated by `flutter pub get`. Both files are git-untracked, so
  they are not part of the diff.
- `app/ios/Podfile.lock` — regenerated by `pod install` in CI's iOS
  build job; the `flutter_local_notifications` Pod drops out.
  (`Podfile.lock` is git-ignored, no on-disk delete needed.)
- Dart call sites under `app/lib/modules/suggestion/`,
  `app/lib/modules/settings/`, and
  `app/test/support/settings_provider.dart` are touched **only** if a
  v12/v13 / v10 API change requires an adjustment. Default expectation:
  no Dart edits for the plus-plugin and `package_info_plus` bumps. The
  `font_awesome_flutter` bump may touch any of ~15 icon call sites only
  if v11 renames a constant currently used by the app.
- No backend (`server/`), `openapi/`, `web/`, CI, or SDK-constraint
  changes. No behavior change for end users — the deleted plugin has
  zero call sites; the bumped plugins keep their public surface or are
  updated in lockstep with their renames.

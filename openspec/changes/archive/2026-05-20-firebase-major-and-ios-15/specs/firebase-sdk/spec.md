## ADDED Requirements

### Requirement: Firebase Dart suite is on the v4-series majors

`app/pubspec.yaml` SHALL declare every Firebase Dart package at the
v4-series major (or later) so the suite shares one consistent native
backend (Firebase iOS SDK v12.x, Android FirebaseBoM 34.x carried by
`firebase_core`).

The required minimum majors are:

- `firebase_core` `^4.0.0`
- `firebase_auth` `^6.0.0`
- `firebase_analytics` `^12.0.0`
- `firebase_crashlytics` `^5.0.0`
- `firebase_messaging` `^16.0.0`

#### Scenario: pubspec declares the v4 suite

- **WHEN** `app/pubspec.yaml` is inspected after the change
- **THEN** `firebase_core` is `^4.x`
- **AND** `firebase_auth` is `^6.x`
- **AND** `firebase_analytics` is `^12.x`
- **AND** `firebase_crashlytics` is `^5.x`
- **AND** `firebase_messaging` is `^16.x`

#### Scenario: lock file reflects the v4 suite

- **WHEN** `app/pubspec.lock` is regenerated by `flutter pub get`
- **THEN** the resolved `firebase_core` version starts with `4.`
- **AND** the resolved versions of `firebase_auth`, `firebase_analytics`,
  `firebase_crashlytics`, and `firebase_messaging` start with their
  declared majors (`6.`, `12.`, `5.`, `16.` respectively)

### Requirement: The Firebase Dart suite stays in lockstep â€” no mixed majors

The five Firebase Dart packages SHALL all share the same FlutterFire
release line. Mixing a v3 `firebase_core` with v4-series siblings (or
vice versa) is not allowed: FlutterFire's native interop assumes one
release line across the suite.

#### Scenario: All five Firebase packages are on the same FlutterFire release line

- **WHEN** the Firebase entries in `app/pubspec.yaml` are inspected
- **THEN** they all belong to the FlutterFire v4 release line
  (`firebase_core ^4` / `firebase_auth ^6` / `firebase_analytics ^12` /
   `firebase_crashlytics ^5` / `firebase_messaging ^16`)
- **AND** no Firebase Dart package is left on its prior major

### Requirement: The bump is behaviour-neutral on the Dart side

The Firebase v4 bump SHALL NOT change runtime behaviour of the app's
existing call sites:

- `Firebase.initializeApp(...)` in `app/lib/main.dart` and in
  `app/lib/modules/firebase/services/notification/background_handler.dart`
- `FirebaseAnalytics.instance`, `FirebaseAnalyticsObserver`,
  `analytics.setUserProperty`, `analytics.logEvent`,
  `analytics.logScreenView`
- `FirebaseCrashlytics.instance.recordFlutterFatalError`,
  `FirebaseCrashlytics.instance.recordError`
- `FirebaseMessaging.instance`, `FirebaseMessaging.onMessage.listen`,
  `getAPNSToken`, `getToken`, `requestPermission`, `RemoteMessage`

Where the FlutterFire v4 CHANGELOG documents a breaking signature for an
API the app actually invokes, the call site SHALL be migrated to the new
signature. The change SHALL NOT introduce speculative refactors of
unaffected call sites.

#### Scenario: flutter analyze is clean after the bump

- **WHEN** `flutter analyze` is run from `app/` after the bump
- **THEN** it reports no new errors or warnings attributable to the
  Firebase Dart suite
- **AND** the Phase 1 E2E smoke suite passes

#### Scenario: flutter test is green after the bump

- **WHEN** `flutter test` is run from `app/` after the bump
- **THEN** all existing unit and widget tests still pass

### Requirement: Android FirebaseBoM is on the v4-line transitively or pinned

The resolved Android FirebaseBoM SHALL be on the line shipped by `firebase_core` v4 (currently the 34.x line), as visible in `./gradlew :app:dependencies`. If `firebase_core` v4's transitive resolution lags that line at apply time, the change SHALL add an explicit `implementation platform('com.google.firebase:firebase-bom:<resolved>')` to `app/android/app/build.gradle`. Otherwise no Android edit is required.

#### Scenario: BoM is current via transitive resolution

- **WHEN** `./gradlew :app:dependencies` is run after the bump
- **THEN** the resolved `com.google.firebase:firebase-bom` is on the
  34.x line (or whatever line `firebase_core` v4 ships at apply time)
- **AND** `app/android/app/build.gradle` is not modified to pin it

#### Scenario: BoM needs explicit pin

- **WHEN** the transitively-resolved BoM lags the line `firebase_core` v4
  is documented to ship with
- **THEN** `app/android/app/build.gradle` adds an explicit
  `implementation platform('com.google.firebase:firebase-bom:<version>')`
  entry under its `dependencies` block
- **AND** the resolved BoM seen by `./gradlew :app:dependencies` matches
  that pin

## ADDED Requirements

### Requirement: Calendar tokens are persisted durably in the database seam
The app SHALL persist each added calendar (its subscription token and identifying
metadata) in the device-local SQLite store (the `@/db` seam), NOT in an ephemeral
in-memory holder. A persisted calendar SHALL survive app restart, process kill, and
JS-cache clear. The ephemeral in-memory scanned-source holder
(`mobile/src/features/calendar-sources/data/scanned-source.ts`) SHALL be removed.

#### Scenario: A persisted calendar survives a restart
- **WHEN** a calendar is added (via QR scan or iCal import) and the app is then restarted,
  process-killed, or its JS cache cleared
- **THEN** the calendar (its token and metadata) is still present on relaunch
- **AND** it was read back from the durable database store, not an in-memory holder

#### Scenario: The data layer does not import the database backend directly
- **WHEN** the user-calendars repository or read hook reads or writes a calendar
- **THEN** it imports `{ db }`, the `userCalendars` table, and any query helpers from `@/db`
- **AND** it does not import `drizzle-orm`, `expo-sqlite`, or the generated API client outside
  the feature's `data/` sublayer

### Requirement: The `user_calendars` schema mirrors the Flutter wire format for importer fidelity
The app SHALL define a `userCalendars` Drizzle table (SQL table `user_calendars`, the sembast
store name verbatim) whose columns mirror the Flutter `UserCalendar.toDbMap()` wire format so
the Phase-09 one-shot importer can write recovered `user_calendars` rows with no data loss and
minimal transformation: `id` TEXT primary key (the sembast record key and upsert identity),
`token` TEXT not-null (the irreplaceable subscription identity), `name` TEXT not-null, nullable
`schoolName` and `schoolId` TEXT, `lastUpdatedAt` and `createdAt` TEXT not-null holding UTC
ISO-8601 strings, and a `visible` boolean column (stored as SQLite integer, default true).

#### Scenario: The schema columns match the importer wire format
- **WHEN** the `userCalendars` table is defined
- **THEN** it has the columns `id` (TEXT primary key), `token`, `name` (TEXT not-null),
  `schoolName`, `schoolId` (TEXT nullable), `lastUpdatedAt`, `createdAt` (TEXT not-null), and
  `visible` (boolean, default true)
- **AND** the date columns hold UTC ISO-8601 strings and `id` (not the token) is the primary key

#### Scenario: A `toDbMap()`-shaped record imports without transformation
- **WHEN** a record in the Flutter `toDbMap()` wire format (id, token, name, nullable
  schoolName/schoolId, UTC ISO-8601 lastUpdatedAt/createdAt, boolean visible) is written into the
  schema
- **THEN** every field is stored verbatim with no parse-and-re-encode and no data loss

### Requirement: The second real migration is generated and committed
The change SHALL add a second real Drizzle migration, generated by `drizzle-kit generate` (driver
`expo`) from `src/db/schema.ts`: a new `000N_*.sql` creating the `user_calendars` table, an
advanced `meta/_journal.json` (a second entry), a new snapshot, and an updated `migrations.js`,
all committed. The startup migration runner (`runMigrations()`) SHALL apply both migrations
unchanged in its wiring.

#### Scenario: The committed bundle contains the user_calendars migration
- **WHEN** the migration bundle is generated and committed
- **THEN** `src/db/migrations` contains a `CREATE TABLE user_calendars` migration and a second
  journal entry referencing it
- **AND** the bundle is committed (a fresh clone has both migrations without running codegen)

#### Scenario: The runner drives the two-migration bundle
- **WHEN** the app starts and `runMigrations()` runs
- **THEN** it drives `migrate(db, migrations)` with the committed bundle (now two entries)
- **AND** the existing migration-runner proof test still passes

### Requirement: Repository over the database seam for the user-calendars store
The feature SHALL expose an async repository over the `@/db` seam with: `findAll` returning all
calendars, `getById(id)` returning one calendar or undefined, `getByToken(token)` returning one
calendar or undefined, an `upsert` that inserts-or-updates by `id` (mirroring the Flutter `put`),
a `remove(id)`, and a `setVisible(id, visible)`. The repository SHALL map stored rows to a domain
type and domain objects to rows through pure mappers, accept a caller-supplied `id` on upsert (so
the importer can supply a recovered id), and SHALL NOT import the database backend directly.

#### Scenario: Upsert inserts a new calendar and updates an existing one by id
- **WHEN** `upsert` is called with a calendar whose `id` does not yet exist
- **THEN** a new row is inserted
- **WHEN** `upsert` is called again with the same `id` and changed fields
- **THEN** the existing row is updated (no duplicate row)

#### Scenario: getByToken resolves a calendar by its subscription token
- **WHEN** `getByToken(token)` is called for a stored calendar
- **THEN** it returns that calendar mapped to the domain type, or undefined if no row has that token

#### Scenario: remove and setVisible operate by id
- **WHEN** `remove(id)` is called
- **THEN** the row for that `id` is deleted and a subsequent `getById(id)` returns undefined
- **WHEN** `setVisible(id, false)` is called
- **THEN** the `visible` column for that row is updated

### Requirement: Row↔domain and server-DTO mappers
The data layer SHALL convert between the stored row type (TEXT ISO-8601 dates, integer boolean)
and a domain type (exposing `Date` timestamps and a boolean `visible`) through pure mappers.
Writing a domain object SHALL normalize each timestamp to a canonical UTC ISO-8601 string;
reading SHALL parse it back to a `Date`. Nullable `schoolName`/`schoolId` SHALL round-trip
including absent values. The data layer SHALL also expose a pure mapper from the server
`CalendarForPublic` DTO to the domain type (mirroring the Flutter `fromCalendarForPublic`).

#### Scenario: A domain calendar round-trips through the mappers
- **WHEN** a domain calendar is mapped to a row and back to a domain calendar
- **THEN** all fields are preserved, the timestamps are equal, the row's date strings are
  canonical UTC ISO-8601, and `visible` is preserved
- **AND** a null `schoolName` or `schoolId` round-trips as absent

#### Scenario: A server calendar DTO maps to the domain type
- **WHEN** the `CalendarForPublic` → domain mapper is given a server DTO
- **THEN** it returns a domain calendar carrying the DTO's id, token, name, schoolName, schoolId,
  dates (parsed to `Date`), and `visible` defaulting to true

### Requirement: Reactive read hook over the database seam
The feature SHALL expose a reactive read hook returning the current list of persisted calendars
that re-renders consumers when the `user_calendars` table changes, built on the seam's
`useLiveQuery` (so feature code does not import the database backend directly).

#### Scenario: The reactive hook reflects the current rows
- **WHEN** a component uses the user-calendars read hook
- **THEN** it receives the current calendars mapped to the domain type
- **AND** the hook is built on `useLiveQuery` re-exported from `@/db`, not a direct
  `drizzle-orm` import

### Requirement: QR-scan and iCal-import success paths persist a durable calendar
A successful QR scan and a successful iCal import SHALL persist a durable `user_calendars` row
(its token plus the calendar metadata resolved from the server) through the repository, replacing
the prior ephemeral in-memory handoff. Persisting SHALL resolve the calendar's metadata from the
server by token (`GET /calendars/by-token/{token}`). A failed persist (token resolution or upsert
failure) SHALL be recorded through the `@/firebase` `recordError` seam and surfaced as an
accessible failure state.

#### Scenario: A successful add persists a durable row
- **WHEN** a QR scan or iCal import succeeds (yielding a token)
- **THEN** the calendar's metadata is resolved from the server by token and a durable
  `user_calendars` row is upserted (no ephemeral in-memory holder is written)

#### Scenario: A failed persist is recorded and surfaced
- **WHEN** the token resolution or the durable upsert throws during an add
- **THEN** the error is recorded through `@/firebase` `recordError`
- **AND** the screen surfaces an accessible failure state (a polite live region with an alert
  role)

### Requirement: The durable store is verified by automated tests under the coverage gate
The unit test suite SHALL prove the durable store: the row↔domain and server-DTO mappers
(round-trip, importer-fidelity verbatim, canonical-UTC normalization, boolean + null handling),
the id wrapper, the repository operations (proven against the `@/db` seam), a restart-simulation
proving a prior write is read back after a fresh store handle, and the persist wiring (success +
failure). These tests SHALL run under the existing `test-mobile` CI job, and the logic paths
(`src/db/**`, `src/features/calendar-sources/data/**`) SHALL meet the enforced K-3 coverage
threshold (90% lines and branches) on landing.

#### Scenario: The durable store meets the coverage threshold
- **WHEN** the change's test suite is run with coverage
- **THEN** all configured thresholds pass (the new `src/db/**` and
  `src/features/calendar-sources/data/**` paths at 90% lines and branches)

#### Scenario: A prior write is read back after a simulated restart
- **WHEN** a calendar is written and then read back through a freshly-constructed store handle /
  repository module
- **THEN** the calendar is returned unchanged (the write-then-read-back durability contract)

### Requirement: On-device restart durability is proven by an E2E flow or an inbox manual pass
The change SHALL prove on-device durability either by a Maestro flow that adds a calendar,
relaunches the app, and asserts the calendar is still present, OR — if no stable post-relaunch
assertion target exists this ship — by an inbox handoff note specifying the manual restart, kill,
and cache-clear verification. Exactly one of these SHALL be present (no silently-skipped axis).

#### Scenario: Durability is proven on-device
- **WHEN** the change lands
- **THEN** either a Maestro relaunch-durability flow exists under `mobile/.maestro/`, or an inbox
  note describes the manual restart/kill/cache-clear verification with steps to verify

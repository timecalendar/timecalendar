## ADDED Requirements

### Requirement: Synced calendar events are persisted durably in the database seam
The app SHALL persist synced server calendar events in the device-local SQLite store
(the `@/db` seam) in a `calendar_events` table, NOT in an ephemeral holder or the
in-memory query cache. The persisted events SHALL survive app restart, process kill, and
JS-cache clear, so the calendar renders offline from the local table.

#### Scenario: Synced events survive a restart and render offline
- **WHEN** a sync completes and the app is then restarted or relaunched with no network
- **THEN** the previously-synced events are still present and render in the calendar views
- **AND** they were read back from the durable database table, not re-fetched

#### Scenario: The data layer does not import the database backend or generated client outside `data/`
- **WHEN** the calendar sync repository, mappers, read hook, or orchestrator runs
- **THEN** it imports `{ db }`, the `calendarEvents` table, and query helpers from `@/db`,
  and the sync mutation / DTO type from `@/api/generated/**`, only within the feature's
  `data/` sublayer
- **AND** it does not import `drizzle-orm`, `expo-sqlite`, or the generated client from a
  `ui/` screen, route, or the feature barrel

### Requirement: The `calendar_events` schema mirrors the Flutter wire format for importer fidelity
The app SHALL define a `calendarEvents` Drizzle table (SQL `calendar_events`, the sembast
store name verbatim) whose columns mirror the Flutter `CalendarEvent.toDbMap()` /
`CalendarEventForPublic` DTO so the Phase-09 one-shot importer can write recovered rows
with no data loss and minimal transformation: `uid` TEXT primary key (the record key and
replace identity), `title` TEXT not-null, `color` and `groupColor` TEXT not-null
(`#RRGGBB` verbatim), `startsAt`/`endsAt`/`exportedAt` TEXT not-null holding UTC ISO-8601
strings, nullable `location` and `description` TEXT, `allDay` boolean (SQLite integer),
`teachers`/`tags`/`fields` JSON-encoded TEXT (`fields` nullable), `type` TEXT not-null
(the `EventTypeEnum` value, stored verbatim — not a checked enum), and `userCalendarId`
TEXT not-null (the parent calendar id).

#### Scenario: The schema columns match the importer wire format
- **WHEN** the schema is defined
- **THEN** every `CalendarEvent.toDbMap()` / `CalendarEventForPublic` field has a column of
  the matching representation (dates as TEXT ISO-8601 UTC, color as TEXT hex, `allDay` as
  a boolean, `teachers`/`tags`/`fields` as JSON TEXT, `type` as TEXT verbatim)
- **AND** `uid` is the primary key and replace identity

#### Scenario: A `toDbMap()`-shaped record imports without lossy transformation
- **WHEN** an importer (or the DTO mapper) writes a record carrying the wire-format values
- **THEN** scalar fields are stored verbatim, dates are already canonical UTC ISO strings,
  `allDay` maps bool→0/1, and `teachers`/`tags`/`fields` are stored as their JSON form
- **AND** no value parsing that could corrupt or drop data occurs on the import path

### Requirement: The third real migration is generated and committed
The app SHALL add a third committed migration creating `calendar_events`, generated by
`drizzle-kit generate` (driver `expo`): a `0002_*.sql`, a third `meta/_journal.json`
entry, an `0002_snapshot.json`, and an updated `migrations.js` importing all three SQL
files. The bundle SHALL be committed (a fresh clone has the migrations without codegen).

#### Scenario: The committed bundle contains the calendar_events migration
- **WHEN** the repository is freshly cloned with no codegen run
- **THEN** `src/db/migrations/` contains the `0002` SQL + snapshot + the third journal
  entry, and `migrations.js` imports `m0000`, `m0001`, and `m0002`

#### Scenario: The runner applies the three-migration bundle
- **WHEN** `runMigrations()` runs at startup
- **THEN** it drives Drizzle's `migrate()` with the committed three-migration bundle
- **AND** `migrate.test.ts` still passes (the runner is unchanged; only the bundle grew)

### Requirement: Sync orchestration fetches over the durable tokens and replaces the local table atomically
The app SHALL provide a sync orchestrator that reads the durable `user_calendars`
subscription tokens, calls `POST /calendars/sync { tokens }` once (batch) over the single
`customFetch` mutator, flattens the returned calendars' events (attaching each parent
`userCalendarId`), maps the DTOs to rows, and replaces the entire `calendar_events` table
with the result. The replace SHALL be transactional (atomic delete-all then bulk-insert)
so a crash mid-replace never leaves a partially-populated table.

#### Scenario: A successful sync replaces the local events
- **WHEN** there are durable tokens and the batch sync returns calendars with events
- **THEN** the orchestrator flattens the events with their parent `userCalendarId`, maps
  them to rows, and replaces all `calendar_events` rows in one transaction (delete-all
  then bulk-insert)
- **AND** the calendar views reactively reflect the new events

#### Scenario: No tokens means no network call
- **WHEN** there are no durable `user_calendars` tokens
- **THEN** the orchestrator is a no-op and issues no sync request (Flutter parity)

#### Scenario: A failed fetch leaves the last-good rows intact
- **WHEN** the batch sync request fails (e.g. offline or a server error)
- **THEN** the existing `calendar_events` rows are unchanged (the drop+replace runs only
  after a successful fetch)
- **AND** the failure surfaces as a recoverable `isError` state, not a thrown crash

### Requirement: Row↔domain and server-DTO mappers isolate the storage format
The app SHALL provide pure mappers in `data/` that convert between the stored row, the
domain `CalendarEvent`, and the server DTO: `rowToCalendarEvent`, `calendarEventToRow`,
and `fromCalendarEventDto(dto, userCalendarId)`. The mappers SHALL normalize every written
date to canonical UTC (`toISOString()`), encode/decode `teachers`/`tags`/`fields` as JSON
defensively (an unparseable value decodes to a safe default, never throws), and map
null↔undefined and boolean at the edge. They SHALL be the only place the generated DTO
type touches the data layer.

#### Scenario: A domain event round-trips through the row mappers
- **WHEN** a `CalendarEvent` is written via `calendarEventToRow` and read via `rowToCalendarEvent`
- **THEN** every field is preserved, dates are canonical UTC, and JSON columns round-trip

#### Scenario: A corrupt JSON column decodes to a safe default
- **WHEN** a `teachers`/`tags`/`fields` column holds an unparseable value
- **THEN** the decode returns a safe default (`[]` / `null`) and the read does not throw

#### Scenario: A server event DTO maps to a row with its parent calendar id
- **WHEN** `fromCalendarEventDto(dto, userCalendarId)` runs over a `CalendarEventForPublic`
- **THEN** it produces a domain event with the DTO values and the supplied `userCalendarId`,
  mirroring the Flutter `CalendarEvent.fromApi`

### Requirement: The events-source seam sources synced events without a consumer change
The app SHALL swap `useCalendarEvents(range)` to source synced `calendar_events` rows
(reactively) merged with the personal-events read, range-filtered, behind the existing
seam signature and the unchanged `CalendarEvent` domain shape. No calendar view consumer
(timeline screen, agenda list) SHALL require a change. The dense-week fixture SHALL no
longer be part of the default runtime merge.

#### Scenario: The calendar renders synced events through the unchanged seam
- **WHEN** synced events exist and a view reads `useCalendarEvents(range)`
- **THEN** it returns the synced events (mapped to `CalendarEvent`) merged with personal
  events, filtered to the range
- **AND** the timeline screen and agenda list render them with no source-related change to
  their code

#### Scenario: The fixture is dev/test-only
- **WHEN** the app runs normally (not a test or dev seed)
- **THEN** `useCalendarEvents` does not include the dense-week fixture in its result

### Requirement: Sync is triggered at startup and by pull-to-refresh with an accessible status surface
The app SHALL trigger a sync fire-and-forget at startup (offline-safe; a failure is
silent at launch and the last-good rows render) and on a user-initiated pull-to-refresh on
the calendar screen. The pull-to-refresh SHALL show an accessible refreshing state and, on
failure, an accessible sync-error with a retry control (a polite live region + status
role + a labeled retry). The calendar screen SHALL stay presentational (it calls the
`data/` hook; it holds no fetch logic).

#### Scenario: Startup sync runs once, fire-and-forget
- **WHEN** the app starts
- **THEN** a sync is kicked once without blocking first paint, and an offline launch shows
  the last-good rows

#### Scenario: Pull-to-refresh re-syncs with accessible status
- **WHEN** the user pulls to refresh on the calendar
- **THEN** a sync runs, an accessible refreshing state is announced, and on failure an
  accessible error + retry is shown (a recoverable state, the timetable still rendering)

### Requirement: Sync failure observability distinguishes recoverable fetch failure from a local write failure
A sync fetch failure (network/server) SHALL be a recoverable `isError` UI state and SHALL
NOT be reported to Crashlytics (mirroring the read-path posture). A failure of the local
`replaceAll` transaction (a SQLite write failure) SHALL be reported through the `@/firebase`
`recordError` seam (a crash-worthy local-persistence failure) in addition to surfacing an
error.

#### Scenario: A fetch failure is not recorded
- **WHEN** the batch sync request rejects
- **THEN** the failure surfaces as `isError` and is not sent to `recordError`

#### Scenario: A local replace-transaction failure is recorded
- **WHEN** the `replaceAll` transaction throws (a local SQLite write failure)
- **THEN** the error is reported through `@/firebase` `recordError` and surfaced

### Requirement: The sync data layer is verified by automated tests under the coverage gate
The sync `data/` layer SHALL be covered by automated tests under the K-3 90% logic gate:
the mappers (round-trip, importer-fidelity verbatim, canonical-UTC, JSON encode/decode +
defensive decode, null/boolean), the repository query shape and the transactional
drop+replace (delete-then-insert inside one transaction), the DTO mapper, the sync-wiring
at the `customFetch` seam (success, no-tokens no-op, fetch-failure → `isError`,
replace-failure → `recordError`), and a restart-simulation reading back a prior write
through a stateful in-memory `@/db` fake. The presentational screen SHALL meet the 70%
floor.

#### Scenario: The sync layer meets the coverage threshold
- **WHEN** `npm test -- --coverage` runs in `mobile/`
- **THEN** the `data/sync/**` logic clears the 90% gate and the suite is green

#### Scenario: The sync wiring is proven at the mutator seam
- **WHEN** the sync-wiring test mocks `src/api/mutator` and drives the real generated
  mutation with a real `QueryClient`
- **THEN** a successful sync calls `replaceAll` with the flattened+mapped events, a
  no-tokens case issues no request, a fetch failure sets `isError` without `recordError`,
  and a replace-transaction failure calls `recordError`

## Why

Phase 2 ([TIM-3](/TIM/issues/TIM-3)) is the dependency-upgrade & maintenance
phase. The B1 audit ([TIM-36](/TIM/issues/TIM-36), Group B — build hygiene)
identified two coupled items in audit slice B4 ([TIM-45](/TIM/issues/TIM-45))
that remain after the board's 2026-05-20 product decision to drop iPhone
5s/6/6+ and iOS 13–14:

- The Firebase Dart suite is one major behind on every package
  (`firebase_core` 3, `firebase_auth` 5, `firebase_analytics` 11,
  `firebase_crashlytics` 4, `firebase_messaging` 15). The FlutterFire v4
  release line is supported and bumps the underlying native SDKs to the
  current Firebase iOS SDK (v12.x) and Android FirebaseBoM (34.x).
- The Firebase iOS SDK v12 line carried by `firebase_core` v4 **requires
  `IPHONEOS_DEPLOYMENT_TARGET = 15.0`**. The repo currently declares iOS 13
  in `Podfile` / `AppFrameworkInfo.plist` and has a stale `10.0` in the
  Xcode project's base build configurations — neither will satisfy v4.

These two items must ship together: bumping the Firebase Dart suite without
raising iOS to 15 will fail `pod install`; raising iOS to 15 without bumping
the suite is product churn with no upgrade payoff. The Android B8 hygiene
slice (`jcenter` removal, Crashlytics Gradle plugin `2.9.9` → `3.0.3`) was
already shipped independently via [PR #48](https://github.com/timecalendar/timecalendar/pull/48)
and is OUT of this change.

## What Changes

- **BREAKING (platform):** Raise the iOS deployment target from `13.0` to
  `15.0`. iPhone 5s / 6 / 6+ (iOS 13–14) can no longer install future builds.
  Board-approved 2026-05-20 (see [TIM-3](/TIM/issues/TIM-3)). The bump
  touches three files — `app/ios/Podfile`, `app/ios/Flutter/AppFrameworkInfo.plist`,
  and `app/ios/Runner.xcodeproj/project.pbxproj` (three `IPHONEOS_DEPLOYMENT_TARGET = 10.0`
  entries in the base Debug / Release / Profile configurations are aligned to
  `15.0`).
- **Bump the Firebase Dart suite in lockstep:** `firebase_core` 3→4,
  `firebase_auth` 5→6, `firebase_analytics` 11→12, `firebase_crashlytics` 4→5,
  `firebase_messaging` 15→16 in `app/pubspec.yaml`.
- **Refresh `app/pubspec.lock` and `app/ios/Podfile.lock`** so the new Dart
  versions and the v12 Firebase iOS SDK resolve cleanly under iOS 15.
- **Verify, do not pin, the Android FirebaseBoM.** `firebase_core` v4's
  Android Gradle plugin pins a current FirebaseBoM (34.x) transitively. The
  Applier confirms via `./gradlew :app:dependencies` and only adds an
  explicit pin in `app/android/app/build.gradle` if the resolved BoM is
  behind the v4 expected line.

Out of scope: any FlutterFire CLI re-run on `firebase_options.dart` (the
generated file is platform-config only and not affected by the suite
bump), any non-Firebase Dart deps, any backend / `openapi/` / Android
changes beyond an optional BoM pin if needed.

## Capabilities

### New Capabilities
- `firebase-sdk`: records that the app uses the FlutterFire v4 series in
  lockstep across `firebase_core`, `firebase_auth`, `firebase_analytics`,
  `firebase_crashlytics`, and `firebase_messaging`, with no version drift
  inside the suite.
- `ios-platform-minimum`: records the iOS deployment-target minimum and
  the consistency requirement across `Podfile`, `AppFrameworkInfo.plist`,
  and the Xcode project base build configurations.

## Impact

- `app/pubspec.yaml` — five Firebase Dart-suite version bumps.
- `app/pubspec.lock` — regenerated by `flutter pub get`.
- `app/ios/Podfile` — `platform :ios, "13.0"` → `"15.0"`; the `post_install`
  hook's `IPHONEOS_DEPLOYMENT_TARGET = "13.0"` → `"15.0"`.
- `app/ios/Flutter/AppFrameworkInfo.plist` — `MinimumOSVersion` `13.0` → `15.0`.
- `app/ios/Runner.xcodeproj/project.pbxproj` — three `IPHONEOS_DEPLOYMENT_TARGET = 10.0`
  entries in the base Debug / Release / Profile configurations → `15.0`.
- `app/ios/Podfile.lock` — regenerated by `pod install --repo-update`.
- `app/android/app/build.gradle` — only if the resolved FirebaseBoM needs
  an explicit pin (verification step, not a default edit).
- No Dart call-site changes are required by current FlutterFire v4
  CHANGELOGs against this repo's Firebase usage — the Applier verifies via
  `flutter analyze` + `flutter test` and only edits if a real breaking
  signature is hit.
- Behaviour: app-runtime behaviour is unchanged. The store-side effect is
  the iOS 13–14 device drop, already board-approved.
